## 团队建设 / 项目任务文档（修订版）

### 1. 项目定位
我们开发一个双端 Web 电商平台（用户端 + 商家端），目标是完整演示核心交易链路，而不是仅展示静态页面。

---

### 2. 修订后的范围定义
**用户端核心：** 商品浏览、购物车、下单、订单管理、售后、评价、账户中心。  
**商家端核心：** 商品管理、订单履约、售后审批、数据看板、账户管理。  
**不做：** 真实支付、营销活动、多商家入驻、直播电商。

---

### 3. 修订后的技术路线
- 后端：Node.js + Express + TypeScript + Prisma + MySQL
- 前端（双端统一）：React + Vite + TypeScript + Tailwind + TanStack Query
- 文档：Swagger/OpenAPI
- 部署：Docker Compose

> 说明：原文档中“商家端 Vite + 原生 JS”与主规格冲突，已统一改为 React + TS，避免双范式维护成本。

---

### 4. 关键风险（更新）
| 风险 | 概率 | 影响 | 缓解策略 |
|---|---:|---:|---|
| 文档与实现不一致 | 4 | 5 | 每周做“需求-接口-页面”三向对齐检查 |
| 登录与重置密码安全不足 | 3 | 5 | 引入重置令牌流程与接口限流 |
| 库存并发导致超卖 | 3 | 5 | 下单/取消/退款全链路事务化与条件更新 |
| 学期内时间冲突 | 5 | 4 | 核心路径优先，非核心能力延后 |

---

### 5. 分工建议（按模块）
| 成员 | 主模块 | 次模块 |
|---|---|---|
| 成员A | 用户端核心流程（列表/详情/购物车/结算） | 评价与个人中心 |
| 成员B | 商家端（商品/订单/售后） | 仪表盘 |
| 成员C | 后端认证与权限、订单状态机 | API 文档与联调支持 |
| 成员D | 数据模型、事务一致性、容器化部署 | 测试与发布脚本 |

---

### 6. 里程碑（建议）
1. M1：环境与脚手架完成（Docker、后端基础、双前端基础）
2. M2：用户端交易主链路可跑通
3. M3：商家端履约与售后可跑通
4. M4：测试补齐、文档验收、演示准备

---

### 7. 会议机制
- 每周至少 2 次短会（同步阻塞问题）
- 每周 1 次评审会（展示增量、确认下一周范围）
- 每次会议必须产出：决议、待办、负责人、截止时间
