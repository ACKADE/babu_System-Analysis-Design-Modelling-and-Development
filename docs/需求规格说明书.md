### 2. 技术栈明细

| 层级 | 选型 | 理由 |
|------|------|------|
| **后端运行时** | Node.js + Express + TypeScript | 轻量、生态成熟、类型安全 |
| **ORM** | Prisma | 类型安全、自动迁移、Prisma Studio 可视化管理 |
| **数据库** | MySQL | 关系型适合电商数据模型（用户-商品-订单关联） |
| **认证** | JWT 双 Token（Access + Refresh） | Refresh Token 服务器侧存储 + 轮换，支持登出与失效控制 |
| **Decimal 处理** | Express 全局序列化中间件 | Prisma Decimal 自动转 Number，避免 `$numberDecimal` 脏数据 |
| **密码处理** | bcryptjs | 纯 JS 实现，Docker 无编译依赖 |
| **输入校验** | zod | schema 定义即类型推导，中间件统一校验 |
| **图片上传** | multer | 本地磁盘存储，5MB 限制，前端 CSS object-fit 统一比例 |
| **API 文档** | swagger-jsdoc + swagger-ui-express | 注解即文档，与代码同步维护 |
| **测试** | Jest + Supertest | API 集成测试，独立测试数据库 |
| **部署** | Docker Compose | 一键启动全栈环境 |
| **用户端前端** | React + Vite + TypeScript | 组件化、HMR 热更新、类型安全 |
| **用户端路由** | React Router v6 | 标准 SPA 路由 |
| **用户端样式** | Tailwind CSS | 原子化 CSS，开发效率高，响应式内置 |
| **用户端数据请求** | TanStack Query + axios | 自动三态管理、去重、后台刷新、竞态防护、乐观更新 |
| **商店端前端** | React + Vite + TypeScript | 与用户端技术统一，共享开发经验 |
| **商店端路由** | React Router v6 | 同用户端 |
| **商店端样式** | Tailwind CSS | 同用户端 |
| **商店端数据请求** | TanStack Query + axios | 同用户端 |

## 第二部分：功能需求

### 一、用户端（9 个功能模块）

---

#### U01 — 商品列表展示（含分类筛选）

**用户故事：** 作为一个用户，我希望在首页看到所有可售商品，支持按分类筛选，并能点击进入详情页。

**验收条件：**

1. 页面加载时调用 `GET /api/products` 获取全部可售商品（`isActive: true`）
2. 左侧边栏展示二级分类树（一级分类可折叠展开，点击二级分类筛选商品）
3. 默认展示"全部"商品
4. 商品以卡片网格布局展示，每张卡片包含：缩略图、商品名、一句话简介、价格、所属分类名
5. 纯前端分类筛选（数据量小，无需后端分页查询参数）
6. 加载过程中显示骨架屏或 loading 指示器
7. 后端无数据时显示"暂无商品"空状态引导
8. 后端返回错误时显示错误提示 + 重试按钮
9. 点击商品卡片跳转至 `/product/:id` 详情页
10. 已下架商品（`isActive: false`）不出现在列表中
11. 顶部搜索框：支持按商品名关键词实时搜索（纯前端 filter）
12. 不支持分页，全量展示

**对应 API：**
- `GET /api/products` — 商品列表
- `GET /api/categories` — 分类树

#### U02 — 商品详情页

**用户故事：** 作为一个用户，我希望点击商品后看到大图、完整描述、价格、库存、所属分类，并能从详情页直接加入购物车。

**验收条件：**

1. 展示商品大图（详情图）、商品名、所属分类、完整描述、价格、库存数量
2. 展示"加入购物车"按钮和数量选择器（默认 1）
3. 点击加购 → 后端校验库存 → 成功反馈"已加入购物车"，并提供 [去购物车] [继续逛逛] 两个去向按钮
4. 若用户未登录，点击"加入购物车"时弹窗提示登录或跳转登录页
5. 若库存不足（quantity > stock），后端返回 400，前端展示"库存不足，仅剩 X 件"
6. 商品不存在（404）显示友好提示页
7. loading / error / 404 三种状态覆盖

**对应 API：** `GET /api/products/:id`

**图片展示策略：**

| 位置 | 容器宽高比 | CSS object-fit | 效果 |
|------|-----------|---------------|------|
| 商品卡片缩略图 | 1:1 | cover | 填满方框，居中裁切 |
| 商品详情页大图 | 4:3 | contain | 完整展示，不裁切留白 |
| 购物车小图 | 1:1 | cover | 与卡片一致 |
| 订单历史小图 | 1:1 | cover | 与卡片一致 |

#### U03 — 购物车管理

**用户故事：** 作为一个已登录用户，我可以将商品加入购物车，在购物车中修改数量、删除商品，并前往结算。

**验收条件：**

1. 未登录时点击"加入购物车" → 弹出登录提示或跳转登录页
2. 已登录后，购物车数据存储在后端，与用户绑定
3. 购物车页面以列表展示：商品缩略图、名称、单价、数量选择器、行小计
4. **添加商品时后端校验：quantity 不能超过 product.stock，否则返回 400 并提示"库存不足，当前仅剩 X 件"**
5. **已有购物车项再叠加时，叠加后的 quantity 也不能超过 stock**
6. 每行可 + / - 修改数量，最少为 1，最大为当前库存
7. 每行可删除（显示确认提示）
8. 底部实时计算并显示总金额
9. 空购物车显示"购物车是空的" + 返回商品列表按钮
10. 商品被商家下架（`isActive: false`）后，购物车中该行标注"已下架"，不可修改数量，只能删除
11. 导航栏购物车链接旁显示红色数量徽标（显示购物车中商品总件数）
12. 点击"去结算"跳转至 `/checkout` 结算页

**购物车下架处理：**

| 状态 | 用户表现 | 处理 |
|------|---------|------|
| 商品在购物车，管理员下架 | 打开购物车看到"已下架"标记 | 禁止修改数量，只能删除 |
| 下架商品仍在购物车，结算时 | 后端过滤下架商品 | 跳过，保留在购物车中 |
| 商品恢复上架 | "已下架"标记消失 | 恢复正常操作 |

**对应 API：**
- `GET /api/cart`
- `POST /api/cart`
- `PUT /api/cart/:itemId`
- `DELETE /api/cart/:itemId`

#### U04 — 结算与模拟支付

**用户故事：** 作为一个用户，我希望确认订单信息并完成支付（模拟），看到支付成功的结果。

**验收条件：**

1. 结算页展示三部分：收货信息表单 / 商品清单 / 总金额
2. 收货信息表单：姓名、地址、手机号，均为必填，手机号须为 7-15 位数字
3. 商品清单展示购物车中有效商品（下架商品自动跳过）
4. **结算校验：后端验证每件商品的 quantity ≤ product.stock，任一件库存不足则返回 400 并提示具体商品名和剩余库存**
5. 点击"提交订单并支付" → 后端校验通过 → **Prisma 事务原子操作：创建订单 + 扣减库存 + 生成订单编号 + 清空已结算购物车项**
6. 前端提交后跳转至支付成功页，展示订单号、商品清单、金额、收货信息
7. 模拟支付：后端直接标记订单状态为 `PAID`，不对接真实支付接口
8. **订单编号格式：YYYYMMDD + 4位序号（如 202606020001），通过 OrderSequence 表原子递增保证并发唯一**
9. 重复提交防护：按钮点击后立即禁用 + loading 动画，防止重复下单

**对应 API：** `POST /api/orders`

#### U05 — 历史订单查看

**用户故事：** 作为一个用户，我希望查看我提交过的所有订单，以及单个订单的详细信息。

**验收条件：**

1. 订单列表页展示：订单号、下单时间、总金额、订单状态，按时间倒序
2. 每条可点击进入订单详情页
3. 订单详情页展示：商品清单（名称、数量、单价）、收货信息、总金额、订单状态

#### U07 — 订单取消与售后（退货退款）

**用户故事：** 作为一个用户，我希望取消尚未发货的订单，或在收到货后申请售后服务（退货退款）。

**验收条件：**

##### U07a — 取消订单（仅退款）

1. 订单详情页，状态为 `PAID` 时显示"取消订单"按钮
2. 点击弹出确认提示："确定取消此订单吗？取消后将自动退款。"
3. 确认后调用 `POST /api/orders/:id/cancel`
4. 后端 Prisma 事务：修改状态为 `CANCELLED` + **恢复库存**（每个 OrderItem 的 quantity 加回 Product.stock）
5. `SHIPPED`、`COMPLETED` 等状态不可取消，返回 400
6. 取消成功后订单详情页显示"已取消"

##### U07b — 申请售后（退货退款）

1. 订单详情页，状态为 `COMPLETED` 且 `returnAttempts < 3` 时显示"申请售后"按钮
2. 点击进入售后申请表单：退货原因（必填，最少 5 字，textarea）
3. 提交后调用 `POST /api/orders/:id/return`，订单状态变为 `RETURN_PENDING`
4. `RETURN_PENDING` 或 `REFUNDED` 状态下不可再申请
5. 售后申请提交后，等待管理员审批
6. 被拒绝后状态回到 `COMPLETED`，`returnAttempts` +1，用户可再次申请（最多 3 次）
7. `returnAttempts >= 3` 时按钮消失，显示"售后次数已达上限（3次）"

##### U07c — 查看售后结果

1. 审批通过：订单状态直接变为 `REFUNDED`（模拟退款 + 恢复库存），终态不可再操作
2. 审批拒绝：状态回到 `COMPLETED`，拒绝原因可查看，`returnAttempts` +1，用户可再次申请（最多 3 次）
3. 第 3 次被拒后 `returnAttempts = 3`，"申请售后"按钮永久消失
4. 退款后（`REFUNDED`）订单状态为终态，不可再操作

**订单完整状态流转：**

```
PAID ──→ CANCELLED（取消）

PAID ──→ SHIPPED ──→ COMPLETED（手动确认收货 / 7天自动确认）
                        │
                        └──→ RETURN_PENDING ──→ REFUNDED（售后退款）
                                 │
                                 └──→ COMPLETED（拒绝退货，returnAttempts +1）
```

**状态标签颜色：**

| 状态 | 颜色 | 说明 |
|------|------|------|
| PAID | 蓝色 | 已支付 |
| SHIPPED | 橙色 | 已发货 |
| COMPLETED | 绿色 | 已完成 |
| CANCELLED | 灰色 | 已取消 |
| RETURN_PENDING | 黄色 | 退货待审批 |
| REFUNDED | 灰色 | 已退款 |

**对应 API：**
- `POST /api/orders/:id/cancel` — 取消订单
- `POST /api/orders/:id/return` — 申请退货（body: returnReason）
- `POST /api/orders/:id/return/approve` — 管理员同意退货
- `POST /api/orders/:id/return/reject` — 管理员拒绝退货（body: rejectReason 可选）

## 第三部分：数据模型

### 1. ER 图（文字描述）

```
Category 1 ── * Category (自引用：父子分类)
Category 1 ── * Product
User 1 ── * CartItem * ── 1 Product
User 1 ── * Order
Order 1 ── * OrderItem * ── 1 Product（通过快照保存，关联但独立）
OrderSequence（独立序号表，原子递增生成订单号）
Order 1 ── * Review * ── 1 User
Product 1 ── * Review
User 1 ── * RefreshToken
User 1 ── * PasswordResetToken
```

### 2. Prisma Schema 设计

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "mysql"
  url      = env("DATABASE_URL")
}

enum UserRole {
  USER
  ADMIN
}

enum OrderStatus {
  PAID
  SHIPPED
  COMPLETED
  CANCELLED
  RETURN_PENDING
  REFUNDED
}

model User {
  id                  Int                   @id @default(autoincrement())
  name                String
  email               String                @unique
  password            String
  role                UserRole              @default(USER)
  createdAt           DateTime              @default(now())
  cartItems           CartItem[]
  orders              Order[]
  reviews             Review[]
  refreshTokens       RefreshToken[]
  passwordResetTokens PasswordResetToken[]
}

model Category {
  id       Int        @id @default(autoincrement())
  name     String
  parentId Int?
  parent   Category?  @relation("CategoryTree", fields: [parentId], references: [id])
  children Category[] @relation("CategoryTree")
  products Product[]

  @@unique([parentId, name])
}

model Product {
  id           Int        @id @default(autoincrement())
  name         String
  summary      String
  description  String     @db.Text
  price        Decimal    @db.Decimal(10, 2)
  stock        Int        @default(0)
  thumbnailUrl String
  imageUrl     String
  isActive     Boolean    @default(true)
  categoryId   Int
  category     Category   @relation(fields: [categoryId], references: [id])
  createdAt    DateTime   @default(now())
  updatedAt    DateTime   @updatedAt
  cartItems    CartItem[]
  orderItems   OrderItem[]
  reviews      Review[]
}

model CartItem {
  id        Int      @id @default(autoincrement())
  userId    Int
  productId Int
  quantity  Int      @default(1)
  createdAt DateTime @default(now())
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  product   Product  @relation(fields: [productId], references: [id], onDelete: Cascade)

  @@unique([userId, productId])
}

model Order {
  id                   Int         @id @default(autoincrement())
  orderNo              String      @unique
  userId               Int
  recipientName        String
  recipientAddress     String
  recipientPhone       String
  totalAmount          Decimal     @db.Decimal(10, 2)
  status               OrderStatus @default(PAID)
  shippedAt            DateTime?
  returnReason         String?     @db.Text
  returnRejectedReason String?     @db.Text
  returnAttempts       Int         @default(0)
  refundedAt           DateTime?
  createdAt            DateTime    @default(now())
  user                 User        @relation(fields: [userId], references: [id], onDelete: Cascade)
  items                OrderItem[]
  reviews              Review[]
}

model OrderItem {
  id           Int     @id @default(autoincrement())
  orderId      Int
  productId    Int
  productName  String
  productPrice Decimal @db.Decimal(10, 2)
  productImage String
  quantity     Int
  order        Order   @relation(fields: [orderId], references: [id], onDelete: Cascade)
  product      Product @relation(fields: [productId], references: [id], onDelete: Restrict)
}

model OrderSequence {
  date    String @id @db.VarChar(8)
  lastSeq Int    @default(0)
}

model RefreshToken {
  id        Int      @id @default(autoincrement())
  userId    Int
  tokenHash String   @unique
  expiresAt DateTime
  revokedAt DateTime?
  createdAt DateTime @default(now())
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
}

model PasswordResetToken {
  id        Int      @id @default(autoincrement())
  userId    Int
  tokenHash String   @unique
  expiresAt DateTime
  usedAt    DateTime?
  createdAt DateTime @default(now())
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
}

model Review {
  id        Int      @id @default(autoincrement())
  userId    Int
  productId Int
  orderId   Int
  rating    Int
  content   String?  @db.Text
  createdAt DateTime @default(now())
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  product   Product  @relation(fields: [productId], references: [id], onDelete: Cascade)
  order     Order    @relation(fields: [orderId], references: [id], onDelete: Cascade)

  @@unique([orderId, productId])
}
```

## 第四部分：API 接口汇总

### 响应约定

**成功响应：** 直接返回数据对象或 `{ message, ...data }`。

**错误响应：** 统一格式：

```json
{
  "message": "人类可读的错误描述",
  "errors": [
    { "field": "body.email", "message": "邮箱格式不正确" }
  ]
}
```

`errors` 数组仅 zod 校验失败时出现，列出每个字段的具体错误。

**HTTP 状态码使用：**

| 状态码 | 含义 |
|--------|------|
| 200 | 操作成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误 / 业务逻辑拒绝（库存不足、状态不允许等） |
| 401 | 未认证或 Token 过期 |
| 403 | 角色无权限（如 USER 访问 ADMIN 接口） |
| 404 | 资源不存在 |
| 413 | 上传文件超过 5MB 限制 |
| 500 | 服务器内部错误 |

## 第五部分：安全与运维要求（补充）

1. **HTTPS**：生产环境强制 HTTPS，禁止明文传输 Token
2. **CORS 白名单**：仅允许用户端/商店端域名访问 API
3. **限流**：登录、忘记密码、重置密码接口启用限流（如 IP 维度）
4. **刷新令牌管理**：Refresh Token 服务器侧存储并轮换，logout 立即失效
5. **管理员操作审计**：商品上下架、订单状态变更、退货审批需记录审计日志（可简化为 console/DB 表）
6. **备份策略**：MySQL 定期备份；上传文件目录进行周期性快照或归档

## 第六部分：总结

- **技术路线：** 全 TypeScript 栈，Express + Prisma + MySQL 后端，React + Vite + Tailwind + TanStack Query 双前端，Docker Compose 部署
- **功能范围：** 用户端 9 模块（浏览、详情、购物车、结算、订单、认证、售后、个人中心、评价），商店端 5 模块（认证、仪表盘、商品管理含分类、订单管理含退货审批、个人中心）
- **数据模型：** 10 张表（User, Category, Product, CartItem, Order, OrderItem, OrderSequence, Review, RefreshToken, PasswordResetToken）
- **API 端点：** 共 31 个（9 认证 + 1 分类 + 5 商品 + 1 仪表盘 + 4 购物车 + 9 订单 + 2 评价）
- **核心设计决策：**
  - 管理员账号由种子数据/已有管理员创建，不开放自助注册
  - 订单商品快照独立存储
  - 上下架软删除、库存扣减带 WHERE stock >= N 并发保护
  - 订单编号原子序号生成（OrderSequence）
  - 二级商品分类（Category 自引用表）
  - 退货退款完整流转（6 状态：PAID/SHIPPED/COMPLETED/CANCELLED/RETURN_PENDING/REFUNDED，最多 3 次申请，仅保留最新一次申请/拒绝原因）
  - 用户确认收货 + 7天惰性自动确认收货
  - 图片文件上传失败回滚清理
  - 前端 CSS object-fit 统一处理图片比例
  - TanStack Query 自动三态管理
  - Prisma Decimal 全局序列化中间件（Decimal → Number）
  - 忘记密码：短期重置令牌 + reset-password 提交
  - Refresh Token 服务端存储 + 轮换，支持登出失效
  - 商品评价：订单完成后可评星+文字，每个订单商品限一条（orderId + productId 唯一）
  - 用户端订单支持 ?status= 状态筛选
  - 管理员订单状态筛选
