# 小型电商平台（Web）需求规格说明书（修订版）

> 说明：本版本对仓库原有方案做了系统修订，重点修正技术路线冲突、安全缺陷、数据模型表达不完整、功能闭环缺失与实施蓝本可执行性不足等问题。

---

## 1. 项目目标与边界

### 1.1 项目目标
构建一个可教学演示、可本地部署、可完整跑通核心电商流程的双端 Web 系统：
- **用户端（Customer）**：浏览商品、购物车、下单、订单追踪、售后、评价。
- **商家端（Admin）**：商品管理、订单履约、售后审批、运营看板。

### 1.2 范围边界
**在范围内：**
- 商品分类与检索
- 购物车与下单
- 模拟支付
- 订单状态流转
- 售后（退货退款）
- 商品评价
- 管理后台（商品、订单、售后、看板）

**暂不纳入：**
- 真实第三方支付
- 营销体系（优惠券/满减/秒杀）
- 多商家入驻
- 复杂推荐系统

## 2. 架构与技术路线（修订后）

### 2.1 总体架构
采用 **模块化单体（Modular Monolith）**，而不是微服务：
- 对课程项目更可控，部署简单，调试成本低。
- 保留清晰模块边界（auth/product/cart/order/review/dashboard），未来可演进为微服务。

### 2.2 技术选型
| 层级 | 选型 | 说明 |
|---|---|---|
| 后端 | Node.js + Express + TypeScript | 生态成熟，开发效率高 |
| ORM | Prisma | 类型安全、迁移与建模效率高 |
| 数据库 | MySQL 8 | 关系数据建模清晰，事务能力强 |
| 前端（双端） | React + Vite + TypeScript | 统一技术栈，降低维护成本 |
| 样式 | Tailwind CSS | 快速构建管理后台与商城页面 |
| 状态请求 | TanStack Query + axios | 服务端状态管理能力强 |
| 文档 | Swagger(OpenAPI) | API 对齐与联调效率高 |
| 容器化 | Docker Compose | 一键本地部署 |

### 2.3 关键技术修正（相较原方案）
1. **修正：商店端“Vite+原生JS”与“React+TS”冲突**  
   统一为 React + TS，避免两套前端范式并存。
2. **修正：账号角色用字符串拼接（USER+ADMIN）不可维护**  
   改为 `Role` + `UserRole` 关系模型，支持多角色、可扩展且查询明确。
3. **修正：忘记密码重置为固定值 123456 属于高风险漏洞**  
   改为一次性重置令牌（TTL）流程。
4. **修正：Decimal 统一转 Number 可能引发金额精度风险**  
   API 层金额字段以字符串返回（如 `"199.90"`），前端做展示格式化。
5. **修正：仅靠“接口读取时惰性自动收货”不稳定**  
   保留惰性检查，同时支持定时任务兜底（Cron/Job）。

## 3. 领域模型与数据设计（修订后）

### 3.1 核心实体
- User（账户）
- Role（角色）
- UserRole（用户-角色映射）
- Category（分类，支持二级）
- Product（商品）
- CartItem（购物车项）
- Order（订单）
- OrderItem（订单快照）
- OrderSequence（订单号序列）
- Review（评价）
- PasswordResetToken（密码重置令牌）

### 3.2 关键设计约束
- 商品下架使用 `isActive` 软下架，不做硬删除。
- 订单商品使用快照字段（name/price/image），防止历史被覆盖。
- 库存扣减必须在事务内执行，且使用条件更新（`stock >= qty`）。
- 订单状态流转必须受状态机约束，不允许跨级跳转。

### 3.3 Prisma 模型建议（关键字段）

```prisma
model User {
  id         Int        @id @default(autoincrement())
  name       String
  email      String     @unique
  password   String
  createdAt  DateTime   @default(now())
  roles      UserRole[]
  cartItems  CartItem[]
  orders     Order[]
  reviews    Review[]
}

model Role {
  id        Int        @id @default(autoincrement())
  code      String     @unique // USER / ADMIN
  users     UserRole[]
}

model UserRole {
  userId    Int
  roleId    Int
  user      User       @relation(fields: [userId], references: [id], onDelete: Cascade)
  role      Role       @relation(fields: [roleId], references: [id], onDelete: Cascade)

  @@id([userId, roleId])
}

model PasswordResetToken {
  id         Int      @id @default(autoincrement())
  userId      Int
  tokenHash   String   @unique
  expiresAt   DateTime
  usedAt      DateTime?
  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

## 4. 功能需求（闭环化修订）

### 4.1 用户端
1. 商品列表/搜索/分类筛选
2. 商品详情（库存校验）
3. 购物车（增删改、失效商品提示）
4. 结算与模拟支付
5. 我的订单（列表、详情、状态筛选）
6. 取消订单（仅 PAID）
7. 确认收货（SHIPPED -> COMPLETED）
8. 售后申请（COMPLETED 可发起，最多 3 次）
9. 商品评价（订单维度唯一）
10. 账户管理（资料、修改密码、退出）

### 4.2 商家端
1. 登录/权限校验
2. 商品管理（创建、编辑、上下架）
3. 订单履约（发货）
4. 售后审批（同意/拒绝）
5. 运营看板（商品数、订单数、销售额、待处理售后）

### 4.3 关键业务规则
- **状态机：**
  - `PAID -> SHIPPED -> COMPLETED`
  - `PAID -> CANCELLED`
  - `COMPLETED -> RETURN_PENDING -> REFUNDED`
  - `RETURN_PENDING -> COMPLETED`（拒绝）
- **终态：** `CANCELLED`、`REFUNDED`
- **并发库存：** 下单、取消、退款三类动作必须事务化。

## 5. API 设计（修订摘要）

### 5.1 鉴权与安全
- Access Token：短时效
- Refresh Token：长时效（建议 HttpOnly + Secure Cookie）
- 所有管理接口必须做服务端 RBAC，不依赖前端守卫

### 5.2 接口分组
- Auth：register/login/refresh/logout/me/profile/password/forgot-password/reset-password
- Products：list/detail/create/update/toggle
- Categories：tree
- Cart：get/add/update/remove
- Orders：create/list/detail/ship/confirm/cancel/return/approve/reject
- Reviews：create/list
- Dashboard：stats

### 5.3 响应约定
- 成功：`{ code: 0, message: "ok", data: ... }`
- 失败：`{ code: <业务码>, message: "错误描述", errors?: [...] }`
- 金额字段统一字符串，避免精度丢失。

## 6. 非功能需求

### 6.1 安全
- 密码 bcrypt 哈希存储
- 登录、重置密码接口限流
- 上传文件白名单 + 大小限制 + 文件名净化
- 关键动作审计日志（发货、退款、角色变更）

### 6.2 稳定性
- 数据库事务覆盖关键写路径
- 全局错误处理中间件
- 健康检查接口 `/healthz`

### 6.3 可观测性
- 结构化日志（请求ID）
- 基础指标（请求耗时、错误率）
