### 2. 技术栈明细

| 层级 | 选型 | 理由 |
|------|------|------|
| **后端运行时** | Node.js + Express + TypeScript | 轻量、生态成熟、类型安全 |
| **ORM** | Prisma | 类型安全、自动迁移、Prisma Studio 可视化管理 |
| **数据库** | MySQL | 关系型适合电商数据模型（用户-商品-订单关联） |
| **认证** | JWT 双 Token（Access + Refresh） | 无状态、Swagger 测试友好、前端角色校验方便 |
| **密码处理** | bcryptjs | 纯 JS 实现，Docker 无编译依赖 |
| **输入校验** | zod | schema 定义即类型推导，中间件统一校验 |
| **图片上传** | multer | 本地磁盘存储，5MB 限制，前端 CSS object-fit 统一比例 |
| **API 文档** | swagger-jsdoc + swagger-ui-express | 注解即文档，与代码同步维护 |
| **测试** | Jest + Supertest | API 集成测试，独立测试数据库 |
| **部署** | Docker Compose | 一键启动全栈环境 |
| **用户端前端** | React + Vite + TypeScript | 组件化、HMR 热更新、类型安全 |
| **用户端路由** | React Router v6 | 标准 SPA 路由 |
| **用户端样式** | Tailwind CSS | 原子化 CSS，开发效率高，响应式内置 |
| **用户端数据请求** | TanStack Query + axios | 自动三态管理、去重、后台刷新、竞态防护、乐观更新 |
| **商店端前端** | React + Vite + TypeScript | 与用户端技术统一，共享开发经验 |
| **商店端路由** | React Router v6 | 同用户端 |
| **商店端样式** | Tailwind CSS | 同用户端 |
| **商店端数据请求** | TanStack Query + axios | 同用户端 |

## 第二部分：功能需求

### 一、用户端（6 个功能模块）

---

#### U01 — 商品列表展示

**用户故事：** 作为一个用户，我希望在首页看到所有可售商品，包括缩略图、名称、一句话简介、价格，并能点击进入详情页。

**验收条件：**

1. 页面加载时调用 `GET /api/products` 获取全部可售商品（`isActive: true`）
2. 商品以卡片网格布局展示，每张卡片包含：缩略图、商品名、一句话简介、价格
3. 加载过程中显示骨架屏或 loading 指示器
4. 后端无数据时显示"暂无商品"空状态引导
5. 后端返回错误时显示错误提示 + 重试按钮
6. 点击商品卡片跳转至 `/product/:id` 详情页
7. 已下架商品（`isActive: false`）不出现在列表中
8. 不支持分页，全量展示

**对应 API：** `GET /api/products`

---

#### U02 — 商品详情页

**用户故事：** 作为一个用户，我希望点击商品后看到大图、完整描述、价格、库存，并能从详情页直接加入购物车。

**验收条件：**

1. 展示商品大图（详情图）、商品名、完整描述、价格、库存数量
2. 展示"加入购物车"按钮，点击后商品进入后端购物车，给出成功反馈
3. 若用户未登录，点击"加入购物车"时弹窗提示登录或跳转登录页
4. 商品不存在（404）显示友好提示页
5. loading / error / 404 三种状态覆盖

**对应 API：** `GET /api/products/:id`

**图片展示策略：**

| 位置 | 容器宽高比 | CSS object-fit | 效果 |
|------|-----------|---------------|------|
| 商品卡片缩略图 | 1:1 | cover | 填满方框，居中裁切 |
| 商品详情页大图 | 4:3 | contain | 完整展示，不裁切留白 |
| 购物车小图 | 1:1 | cover | 与卡片一致 |
| 订单历史小图 | 1:1 | cover | 与卡片一致 |

---

#### U03 — 购物车管理

**用户故事：** 作为一个已登录用户，我可以将商品加入购物车，在购物车中修改数量、删除商品，并前往结算。

**验收条件：**

1. 未登录时点击"加入购物车" → 弹出登录提示或跳转登录页
2. 已登录后，购物车数据存储在后端，与用户绑定
3. 购物车页面以列表展示：商品缩略图、名称、单价、数量选择器、行小计
4. 每行可 + / - 修改数量，最少为 1
5. 每行可删除（显示确认提示）
6. 底部实时计算并显示总金额
7. 空购物车显示"购物车是空的" + 返回商品列表按钮
8. 商品被商家下架（`isActive: false`）后，购物车中该行标注"已下架"，不可修改数量，只能删除
9. 点击"去结算"跳转至 `/checkout` 结算页

**购物车下架处理：**

| 状态 | 用户表现 | 处理 |
|------|---------|------|
| 商品在购物车，管理员下架 | 打开购物车看到"已下架"标记 | 禁止修改数量，只能删除 |
| 下架商品仍在购物车，结算时 | 后端过滤下架商品 | 跳过，保留在购物车中 |
| 商品恢复上架 | "已下架"标记消失 | 恢复正常操作 |

**对应 API：**
- `GET /api/cart` — 获取购物车
- `POST /api/cart` — 添加商品
- `PUT /api/cart/:itemId` — 修改数量
- `DELETE /api/cart/:itemId` — 移除商品

---

#### U04 — 结算与模拟支付

**用户故事：** 作为一个用户，我希望确认订单信息并完成支付（模拟），看到支付成功的结果。

**验收条件：**

1. 结算页展示三部分：收货信息表单 / 商品清单 / 总金额
2. 收货信息表单：姓名、地址、手机号，均为必填，空则拦截
3. 商品清单展示购物车中有效商品（下架商品自动跳过）
4. 点击"提交订单并支付" → 后端校验购物车中至少有一件有效商品 → 创建订单 → 清空已结算的购物车项 → 返回订单号
5. 前端提交后跳转至支付成功页，展示订单号、商品清单、金额、收货信息
6. 模拟支付：后端直接标记订单状态为 `PAID`，不对接真实支付接口
7. 重复提交防护：按钮点击后立即禁用，防止重复下单

**对应 API：** `POST /api/orders`

---

#### U05 — 历史订单查看

**用户故事：** 作为一个用户，我希望查看我提交过的所有订单，以及单个订单的详细信息。

**验收条件：**

1. 订单列表页展示：订单号、下单时间、总金额、订单状态，按时间倒序
2. 每条可点击进入订单详情页
3. 订单详情页展示：商品清单（名称、数量、单价）、收货信息、总金额、订单状态
4. 订单中的商品数据为下单时快照，不受后续商品修改/下架影响
5. 空订单列表显示"还没有订单" + 去逛逛按钮
6. loading / error 状态覆盖

**对应 API：**
- `GET /api/orders` — 当前用户订单列表
- `GET /api/orders/:id` — 单个订单详情

---

#### U06 — 注册与登录

**用户故事：** 作为一个新用户，我希望注册一个账号，以便使用购物车和下单。作为一个已有账号的用户，我希望登录后系统记住我的身份。

**验收条件：**

1. 注册表单：用户名、邮箱、密码、确认密码、隐藏字段标记角色为 USER
2. 登录表单：邮箱 + 密码（不区分角色）
3. 邮箱格式校验、密码最少 6 位、两次密码一致
4. 注册成功后自动登录，返回 JWT
5. 登录成功后导航回用户原本浏览的页面或首页
6. 登录/注册失败显示明确错误信息（如"该邮箱已注册"、"密码错误"）
7. 导航栏根据登录状态切换：未登录显示"登录"，已登录显示用户名 + "退出"
8. 退出后前端清除所有缓存数据（包括 TanStack Query 缓存）

**对称注册逻辑（与商店端共用同一认证接口）：**

| 场景 | 处理 |
|------|------|
| 邮箱不存在，用户端注册 | 创建用户，角色 = USER |
| 邮箱不存在，商店端注册 | 创建用户，角色 = ADMIN |
| 邮箱已存在为 USER，在商店端注册 | 密码验证通过 → 角色追加为 USER+ADMIN |
| 邮箱已存在为 ADMIN，在用户端注册 | 密码验证通过 → 角色追加为 USER+ADMIN |
| 邮箱已存在且目标角色已具有 | 返回提示"该账号已在此端注册过，请直接登录" |
| 邮箱已存在但密码不匹配 | 返回提示"该邮箱已注册，密码错误" |

**对应 API：**
- `POST /api/auth/register` — 注册（body 含 role 字段）
- `POST /api/auth/login` — 登录
- `POST /api/auth/refresh` — Token 续期
- `POST /api/auth/logout` — 登出

### 二、商店端（3 个功能模块）

---

#### S01 — 注册与登录

**用户故事：** 作为一个商店运营者，我通过商店端注册账号并登录，以便管理商品和订单。

**验收条件：**

1. 注册表单：用户名、邮箱、密码、确认密码、隐藏字段标记角色为 ADMIN
2. 登录表单：邮箱 + 密码
3. 注册/登录逻辑与用户端共用同一套后端接口，仅在角色标记上不同
4. 登录成功后跳转至商品管理页
5. 退出后清除所有缓存数据
6. 商店端路由守卫：非 ADMIN 角色用户无法访问商店端任何页面，重定向至登录页
7. 普通用户账号（仅有 USER 角色）通过商店端登录时后端返回 403

**对应 API：** 同 U06，与用户端共用认证接口

---

#### S02 — 商品管理

**用户故事：** 作为一个商店运营者，我希望管理所有商品，包括添加、编辑、上架/下架商品。

**验收条件：**

1. 商品列表页以表格展示所有商品（含已下架），列：缩略图、名称、价格、库存、一句话简介、状态标签（上架/下架）
2. 添加商品按钮 → 打开表单页
3. 添加/编辑表单包含字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| 商品名称 | text input | 必填 |
| 一句话简介 | text input | 卡片列表展示的文字简介 |
| 详情描述 | textarea | 详情页展示的完整描述 |
| 价格 | number input | 必填，须 > 0 |
| 库存 | number input | 必填，须 >= 0 |
| 缩略小图 | file upload | 列表卡片用，建议方图 |
| 详情大图 | file upload | 详情页主图 |

4. 编辑表单预填已有数据，修改后保存
5. 点击上架/下架切换按钮，数据库中切换 `isActive` 状态
6. 下架后用户端不再展示该商品
7. 不支持硬删除商品（保留订单关联数据的完整性）
8. 图片上传预览，上传中 loading 状态
9. 操作完成后 Toast 反馈（创建成功 / 编辑成功 / 上下架成功）

**图片上传规则：**
- 单文件上限 5MB（multer `limits.fileSize`）
- 一次最多上传 2 个文件（缩略图 + 大图）
- 文件名自动加时间戳前缀，防止覆盖
- 超出大小限制 → 后端返回 413，前端提示"图片最大 5MB"
- 图片比例不限，由前端 CSS `object-fit` 统一处理视觉效果

**对应 API：**
- `GET /api/products` — 全部商品（含已下架，管理员视图）
- `POST /api/products` — 创建商品
- `PUT /api/products/:id` — 编辑商品
- `PATCH /api/products/:id/toggle` — 上架/下架切换

---

#### S03 — 订单管理

**用户故事：** 作为一个商店运营者，我希望查看所有用户提交的订单，以及每个订单的详细信息，并能更新订单发货状态。

**验收条件：**

1. 订单列表以表格展示：订单号、用户名、总金额、状态标签、下单时间，按时间倒序
2. 状态标签颜色区分：PAID（蓝色）、SHIPPED（橙色）、COMPLETED（绿色）
3. 点击某条订单进入详情页
4. 订单详情展示：订单号、用户名、商品清单（名称、数量、单价）、收货信息、总金额、当前状态
5. 状态更新：下拉选择或按钮切换新状态，单向流转：PAID → SHIPPED → COMPLETED
6. 状态不可逆，已完成的订单不能回退
7. 空列表显示"暂无订单"
8. loading / error 状态覆盖

**订单状态流转：**

```
PAID（已支付） → SHIPPED（已发货） → COMPLETED（已完成）
    ↓                ↓                    ↓
  蓝色标签          橙色标签            绿色标签
```

**对应 API：**
- `GET /api/orders` — 全部订单（ADMIN 角色）
- `GET /api/orders/:id` — 单个订单详情
- `PATCH /api/orders/:id/status` — 更新订单状态

## 第三部分：数据模型

### 1. ER 图（文字描述）

```
User 1 ── * CartItem * ── 1 Product
User 1 ── * Order
Order 1 ── * OrderItem * ── 1 Product（通过快照保存，关联但独立）
```

### 2. Prisma Schema 设计

```prisma
model User {
  id        Int        @id @default(autoincrement())
  name      String
  email     String     @unique
  password  String
  role      String     @default("USER") // "USER" | "ADMIN" | "USER,ADMIN"
  createdAt DateTime   @default(now())
  cartItems CartItem[]
  orders    Order[]
}

model Product {
  id           Int        @id @default(autoincrement())
  name         String
  summary      String     // 一句话简介（卡片展示）
  description  String     @db.Text  // 详情描述
  price        Decimal    @db.Decimal(10, 2)
  stock        Int        @default(0)
  thumbnailUrl String     // 缩略小图路径
  imageUrl     String     // 详情大图路径
  isActive     Boolean    @default(true)
  createdAt    DateTime   @default(now())
  updatedAt    DateTime   @updatedAt
  cartItems    CartItem[]
  orderItems   OrderItem[]
}

model CartItem {
  id        Int      @id @default(autoincrement())
  userId    Int
  productId Int
  quantity  Int      @default(1)
  createdAt DateTime @default(now())
  user      User     @relation(fields: [userId], references: [id])
  product   Product  @relation(fields: [productId], references: [id])

  @@unique([userId, productId])
}

model Order {
  id               Int         @id @default(autoincrement())
  userId           Int
  recipientName    String      // 收货人姓名
  recipientAddress String      // 收货地址
  recipientPhone   String      // 收货电话
  totalAmount      Decimal     @db.Decimal(10, 2)
  status           String      @default("PAID") // PAID | SHIPPED | COMPLETED
  createdAt        DateTime    @default(now())
  user             User        @relation(fields: [userId], references: [id])
  items            OrderItem[]
}

model OrderItem {
  id           Int     @id @default(autoincrement())
  orderId      Int
  productId    Int
```

## 第五部分：总结

- **技术路线：** 全 TypeScript 栈，Express + Prisma + MySQL 后端，React + Vite + Tailwind + TanStack Query 双前端，Docker Compose 部署
- **功能范围：** 用户端 6 模块（浏览、详情、购物车、结算、订单、认证），商店端 3 模块（认证、商品管理、订单管理）
- **核心设计决策：** 对称注册角色叠加、订单商品快照独立存储、上下架软删除、前端 CSS 统一处理图片比例
